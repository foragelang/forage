# CLI reference

The `forage` binary is the entry point for everything you do at the
shell. Five subcommands, in roughly the order you reach for them:

| Command | What it does |
|---|---|
| [`forage run`](#forage-run) | Parse a recipe, run it, print the snapshot. |
| [`forage capture`](#forage-capture) | Open a real browser, record every fetch / XHR. |
| [`forage scaffold`](#forage-scaffold) | Convert a captures JSONL into a starter recipe. |
| [`forage test`](#forage-test) | Run a recipe against fixtures, diff against a golden snapshot. |
| [`forage publish`](#forage-publish) | Push a recipe to the Forage hub. |

Run `forage --help` or `forage help <subcommand>` at any time for the
argument-parser autogenerated reference. This page is the prose tour.

## `forage run`

Run a `.forage` recipe end-to-end and print the snapshot as JSON.

```sh
forage run <recipe> [--input k=v]...
```

`<recipe>` is either a path to a `.forage` file or a directory
containing `recipe.forage`. The engine is selected from the recipe's
`engine http | engine browser` declaration — `run` doesn't ask.

### Options

- `--input k=v` — bind a recipe input. Repeatable. Values are parsed
  as: `true` / `false` / `null` literals; integers; doubles; JSON
  literals when the value starts with `[` or `{`; everything else is
  a string.
- `--mfa` / `--no-mfa` — toggle stdin-based MFA prompting for recipes
  whose `auth.session.<...>` block declares `requiresMFA: true`.
  Defaults to on; `--no-mfa` makes such recipes fail with
  `auth-mfa-cancelled` rather than blocking on input. See
  [Authenticated sessions](/docs/auth-sessions) for the full session-auth
  contract — including where credentials come from
  (`FORAGE_SECRET_<NAME>` env vars) and how re-auth on 401/403 works.

### Output

- **stdout**: pretty-printed `Snapshot` JSON.
- **stderr**: validation warnings (if any), `stallReason` line,
  diagnostic report sections.

Exit codes: `0` on success, `1` on parse / validation / runtime
failure.

### Examples

```sh
# Run a Wikipedia recipe with a topic input
forage run recipes/wikipedia --input topic=Foraging

# Browser-engine recipe with multiple inputs
forage run recipes/jane \
    --input dispensarySlug=trilogy \
    --input dispensaryName="Trilogy Wellness" \
    --input siteOrigin=https://trilogy.health \
    --input menuPath=/shop/adult-use-menu \
    --input menuType=RECREATIONAL
```

## `forage capture`

Launch an embedded WKWebView, navigate to a URL, record every fetch /
XHR exchange as JSONL.

```sh
forage capture <url> [--out PATH] [--settle SECONDS] [--timeout SECONDS]
```

The injected JS wrapper patches `window.fetch` and `XMLHttpRequest`
in every frame. Each request / response is forwarded to the Swift
host and appended to the output file. The session ends when the
network has been quiet for `--settle` seconds, or `--timeout` seconds
elapse — whichever comes first.

### Options

- `--out PATH` — output JSONL path. Default: `/tmp/probe-captures.jsonl`.
- `--settle SECONDS` — idle seconds before declaring the page "done."
  Default: `8`.
- `--timeout SECONDS` — hard ceiling. Default: `60`.

### Output

Each line of the JSONL file is one capture:

```json
{
  "timestamp": "2026-05-11T12:00:00Z",
  "kind": "fetch",
  "method": "GET",
  "requestUrl": "https://api.example.com/products?page=1",
  "responseUrl": "https://api.example.com/products?page=1",
  "requestBody": "",
  "status": 200,
  "bodyLength": 12340,
  "body": "{\"products\":[…]}"
}
```

### Example

```sh
# Capture a dispensary's menu page; let JS run for 15 seconds idle.
forage capture https://example-dispensary.com/menu \
    --out /tmp/example.jsonl \
    --settle 15
```

## `forage scaffold`

Turn a captures JSONL into a starter `.forage` recipe.

```sh
forage scaffold <captures.jsonl> [--host SUBSTRING] [--out PATH]
```

The heuristics, in order:

1. Group captures by **structural URL pattern**: lowercase host +
   path with numeric and ≥8-char hex segments collapsed to `:id`,
   query string stripped.
2. The **dominant pattern** (most captures) wins as the items
   endpoint; `--host` substring filter narrows the candidate set
   first.
3. For the dominant group, decode JSON bodies and find the **biggest
   homogeneous array of objects** — the array whose elements share
   the most field names.
4. Walk the inferred item shape; per-key type inference:
    - `String` for `*[Ii]d`, `name`, `title`, `description`, `slug`,
      `url`, `*[Uu]rl`, `sku`, `_id$`.
    - `Double` for `price`, `*[Pp]rice`, numeric values with decimals.
    - `Int` for pure integer values.
    - `Bool` for keys ending `able`, `is_*`, `has_*`, or boolean values.
    - `[String]` for arrays of strings.
    - Nested objects are skipped with a `// TODO:` marker — the
      recipe author handles those by hand.
5. **Engine selection**: `application/json` content-type on the
   dominant pattern → `engine http` + `step + paginate.untilEmpty`.
   `text/html` or no clear API → `engine browser` + `captures.match`.

The output recipe always declares the inputs `dispensarySlug`,
`dispensaryName`, and `siteOrigin`, a single `Product` type, an emit
block with field bindings, and one `expect` clause.

### Options

- `--host SUBSTRING` — restrict to captures whose host contains this
  substring. Useful when the page hit multiple analytics / ad
  domains alongside the real API.
- `--out PATH` — write to this path instead of stdout.

### Example

```sh
forage capture https://shop.example.com --out shop.jsonl
forage scaffold shop.jsonl --host shop.example.com > recipes/shop/recipe.forage
$EDITOR recipes/shop/recipe.forage
```

The result is a 50%-right skeleton — the URL pattern, type shape,
and engine choice are inferred from real traffic; pagination
params, auth headers, and any post-processing transforms are the
recipe author's job to fill in.

## `forage test`

Run a recipe against fixtures and diff the produced snapshot
against an expected golden file.

```sh
forage test <recipe-dir> [--update]
```

### Directory layout

```
recipes/<name>/
    recipe.forage
    fixtures/
        captures.jsonl       # browser engine: response bodies to replay
        inputs.json          # optional: { "menuType": "RECREATIONAL", ... }
    expected.snapshot.json   # optional: golden file
```

The test command:

1. Reads `recipe.forage`, parses + validates it.
2. Reads `fixtures/inputs.json` for input bindings.
3. Reads every `.jsonl` file under `fixtures/`:
    - For browser-engine recipes — replays captures via `BrowserReplayer`.
    - For HTTP-engine recipes — registers each line as an `HTTPReplayer`
      fixture.
4. Runs the recipe through the appropriate replay-mode engine.
5. If `expected.snapshot.json` exists, compares records as
   `(typeName, fields)` set-equality (ordering ignored).
6. Surfaces any `unmetExpectations` from the diagnostic report.

### Options

- `--update` — write the produced snapshot to
  `expected.snapshot.json`. The first run on a new fixture set is
  typically `forage test recipes/foo --update`; subsequent runs
  without `--update` are the regression gate.

### Exit codes

- `0` — snapshot matched (or no `expected.snapshot.json` exists)
  AND no unmet expectations.
- `1` — snapshot mismatch OR unmet expectations.
- `2` — setup error: missing recipe / bad parse / validation
  failure / run error.

### Examples

```sh
# Pin the current behavior as the golden:
forage test recipes/sweed --update

# Later, after editing the recipe — see what changed:
forage test recipes/sweed
```

When the snapshot drifts:

```text
$ forage test recipes/sweed
snapshot mismatch: 1 missing, 1 extra
  - missing: Product { externalId=42, name=Old Name, price=12.99 }
  + extra:   Product { externalId=42, name=New Name, price=12.99 }
```

## `forage publish`

Push a recipe to the Forage hub at `https://api.foragelang.com`.

```sh
forage publish <recipe-dir>
```

**Status:** stub for now. The CLI validates the recipe and prints
the JSON payload it would POST. Once the hub lands (Milestone 4 in
the roadmap) it'll actually publish.

### Environment

- `FORAGE_HUB_URL` — override the hub base URL. Default
  `https://api.foragelang.com`.
- `FORAGE_HUB_TOKEN` — API key. Without it, publish prints what it
  would send and exits.
